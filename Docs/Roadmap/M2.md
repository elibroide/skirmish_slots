## PRIORITY TIER 1: CRITICAL - "Make it Look Like a Real Game"
*Without these, the game looks too blocky and playtesters won't engage*

### 1.1 üé® Core Animation System (Foundation)
**Why Critical:** Animations provide clarity and make actions feel impactful. Without them, players don't know what's happening.

**Implementation Approach:**
Since ALL animations are critical but time isn't pressured, implement in phases:

**Phase 1A: Essential Clarity Animations (Week 1-2)**
These make the game UNDERSTANDABLE:
- [ ] **Unit Deployed**
  - Card slides from hand ‚Üí slot with smooth arc
  - Unit "enters" with scale-up animation (0.3s)
  - Deploy effects trigger immediately after
  
- [ ] **Unit Takes Damage**
  - Flash red (0.2s)
  - Shake/recoil effect
  - Damage number floats up: "-3" in red text (fades out in 1s)
  
- [ ] **Shield Damaged/Depleted**
  - Shield icon glows when hit
  - Cracks appear as shield depletes
  - Shatters when fully gone (0.4s animation)
  
- [ ] **Unit Dies**
  - Fade out or disintegrate effect (0.5s)
  - Card slides to graveyard pile
  - Death effects trigger during fade
  
- [ ] **Unit Buffed**
  - Flash green or gold (0.2s)
  - Sparkle/glow effect
  - Buff value appears: "+2 Power" in green (fades in 1s)
  
- [ ] **Action Card Played**
  - Card moves from hand to center screen
  - Brief expansion to show effect (0.4s)
  - Effect resolves (with targeting line if applicable)
  - Card slides to graveyard

**Phase 1B: Game Flow Animations (Week 2-3)**
These make the game flow feel COMPLETE:
- [ ] **Cards Dealt (Round Start)**
  - Cards fly from deck ‚Üí hand one by one (0.1s each)
  - Fanned out animation in hand
  
- [ ] **Round/Skirmish End**
  - Power comparison animation (numbers pulse)
  - Winner announcement per lane (green glow for won lanes)
  - Banner slides in: "Player 1 Wins Skirmish 1" (2s display)
  - Board clears (cards slide to graveyard)
  
- [ ] **Game End**
  - Final score animation
  - Winner celebration (confetti, glow, etc.)
  - "Player 1 Wins 2-1" banner (stays until dismissed)

**Phase 1C: Advanced Animations (Week 3-4)**
These add JUICE and make specific effects satisfying:
- [ ] **Projectile Attacks** (Archer, Turret, etc.)
  - Arrow/bolt fires from unit ‚Üí target (0.3s travel)
  - Impact flash on target
  - Damage numbers appear
  
- [ ] **Unit Bounced**
  - Arc animation returning to hand (0.5s)
  - Whoosh effect
  
- [ ] **Slot Modified** (Ongoing Effect Added)
  - Glow effect pulses on slot (0.3s)
  - Icon appears showing effect type
  
- [ ] **Cleanse Effect**
  - Wipe/flash across unit (0.3s)
  - Keyword icons disappear
  
- [ ] **Leader Ability Activated**
  - Portrait glows/pulses (0.4s)
  - Visual effect (spark, aura, etc.)
  - If targeted, targeting line appears
  
- [ ] **Triggered Abilities** (On-board units)
  - Quick flash on triggering unit (0.2s)
  - Effect animation (specific to ability)
  - Example: Turret ‚Üí muzzle flash ‚Üí projectile ‚Üí impact

**Animation System Requirements:**
- [ ] **Choose & Setup Animation Library**
  - Recommended: Framer Motion (React-friendly, good sequencing)
  - Alternative: React Spring (more control, steeper learning)
  
- [ ] **Animation Queue System**
  - Queue all game events: [Play card, Deal damage, Unit dies, etc.]
  - Process sequentially (wait for animation before next)
  - Allow fast-forward/skip (hold spacebar?)
  
- [ ] **Timing Control**
  - Control WHEN damage appears (sync with impact animation)
  - Example: Fireball ‚Üí travel ‚Üí explosion ‚Üí show damage numbers
  
- [ ] **Customizable Assets per Source**
  - Map effect type ‚Üí animation variant
  - Turret damage ‚â† Wolf damage (different projectiles)
  - Build asset library over time

**Success Metric:** Playtesters say "it feels satisfying to play cards" and "I can follow what's happening"

### 1.2 üé¥ Unit Art & Visual Identity
**Why Critical:** Blocky placeholder cards look amateur. Real art makes the game feel legitimate.

- [ ] **Cel-Shaded Art Style**
  - Establish style guide (color palette, line weight, shading approach)
  - Test with 3-5 cards first to validate style
  
- [ ] **Unit Stories ‚Üí AI Art**
  - Write short story/description for each unit (3-5 sentences)
  - Example: "A grizzled veteran wielding a tower shield, she plants herself at the front line and refuses to budge. Her armor is battered but her resolve is unbreakable."
  - Feed to AI art generator (Midjourney, DALL-E, etc.)
  - Iterate on prompts to match cel-shaded style
  
- [ ] **Multi-Word Unit Names**
  - Replace generic names: "Scout" ‚Üí "Emberwatch Scout"
  - Names hint at abilities: "Stoneheart Guardian" = defensive
  - 2-3 words per name
  
- [ ] **Card Frame Design**
  - Professional card frame template (borders, text boxes, stat areas)
  - Rarity-colored borders (Bronze = copper, Silver = silver, Gold = gold)
  - Clear stat display: Power/Shield in large numbers

**Production Pipeline:**
1. Prioritize cards by usage frequency (most-played cards get art first)
2. Start with ~20 core cards for initial playtests
3. Add more art as testing reveals which cards are keepers
4. Placeholders OK for rarely-played cards initially

**Success Metric:** Playtesters say "this looks like a real card game" and cards are visually distinct

### 2.1 üëë Leader System (Proven Mechanic)
**Why Important:** You've already tested this - it works. Adds consistent strategic value.

- [ ] **Leader Selection in Deck Build**
  - Choose 1 leader before game starts
  - Leader stored separately from deck
  
- [ ] **Leader Object Structure**
  ```javascript
  {
    name: "Commander Valeria",
    art: "valeria_portrait.png",
    ability: "Once per turn: Give a slot 'Deploy: +1 Power'",
    abilityType: "activated", // or "passive"
    maxCharges: 3,
    currentCharges: 3,
    chargesResetPerSkirmish: false // or true, TBD
  }
  ```
  
- [ ] **Quick Action System**
  - Leader abilities don't auto-pass turn
  - Player can activate multiple times in one turn (if charges available)
  - Integrate with Pass/Done button (1.9)
  
- [ ] **Charge Management**
  - Spend charges when ability activated
  - Track per skirmish or per match (decide based on playtesting)
  - UI updates immediately (see 1.7)
  
- [ ] **Example Leaders to Implement (Start with 3-5)**
  - Aggressive Leader: "Activated (1 charge): Deal 2 damage to an enemy"
  - Defensive Leader: "Passive: Your units with Shield get +1 Shield"
  - Tactical Leader: "Activated (1 charge): Move a unit to a close empty slot"

**Design Questions to Answer in Playtesting:**
- Do charges reset per skirmish or stay for whole match?
- How many charges is balanced? (3? 4? 5?)
- Should some leaders have more charges but weaker abilities?
- Are passive abilities as fun as activated?

**Success Metric:** Playtesters use leader abilities strategically, games feel more dynamic

### 2.2 üõ°Ô∏è Shield Mechanic
**Why Important:** Adds durability without just buffing power. Creates tactical decisions (target shielded vs non-shielded).

- [ ] **Shield as Unit Stat**
  - Units have Power/Shield format: "5/2" = 5 power, 2 shield
  - Displayed clearly on cards (both frames)
  
- [ ] **Damage Reduction Logic**
  ```javascript
  // When unit takes damage:
  if (unit.shield >= damage) {
    unit.shield -= damage;
    // Power unaffected
  } else {
    const overflow = damage - unit.shield;
    unit.shield = 0;
    unit.power -= overflow;
  }
  ```
  
- [ ] **Shield Granting**
  - Units can grant shield: "Deploy: Shield 2" (self), "Give close allies Shield 1"
  - Actions can grant shield: "Give an ally Shield 3"
  
- [ ] **Visual System**
  - Shield icon with value on unit card
  - Shield bar or overlay on board frame
  - Animation when shield takes damage (see 1.1)

**Cards to Implement:**
- Shielded units (varying power/shield ratios)
- Shield-granting support units
- Anti-shield cards? ("Ignore shield when dealing damage")

**Success Metric:** Playtesters understand shield vs power, make targeting decisions based on shield

- [ ] **Dominant Keyword**

- [ ] **Lead Detection**

- [ ] **Visual Indication**

- [ ] **Example Dominant Effects**

**Cards to Implement:**

- Test which effects are too weak (ignored)

**Success Metric:** Playtesters fight for control of specific lanes to activate Dominant

### 2.4 üèÜ Conquer Rework
**Why Important:** Simplifies resolution, makes high-value units obvious. Adds bonus SP without card draw snowball.

- [ ] **New Conquer Definition**
  - Conquer is a UNIT ability (keyword on the unit)
  - When a unit with Conquer wins its lane, the player gets +1 SP (in addition to the base 1 SP for winning)
  - Default: Win lane = 1 SP
  - With Conquer unit: Win lane = 2 SP (base 1 + Conquer 1)
  
- [ ] **Conquer Keyword Display**
  - "Conquer" text on unit card
  - Flag/trophy icon in keyword symbols (see 1.4)
  - Tooltip: "If this unit wins its lane, gain +1 SP"
  
- [ ] **Visual Indication**
  - Units with Conquer show flag/trophy icon on board frame
  - When counting SP at round end, clearly show Conquer bonuses
  - Example: "Lane 2: Won (+1 SP) + Conquer (+1 SP) = 2 SP"
  
- [ ] **SP Calculation**
  - At round end: For each lane won, check if winning unit has Conquer
  - Base SP + Conquer SP = Total SP for that lane
  - Display in scoreboard (see 1.8)

**Cards to Implement:**
- Units with Conquer at different rarities
  - Bronze: Lower power units with Conquer (e.g., 3 power + Conquer)
  - Silver: Medium power with Conquer + other ability
  - Gold: High power with Conquer + strong effect
- Test if Conquer units become auto-includes (too strong)
- Test if Conquer is worth the deckbuilding slot

**Design Questions:**
- Can multiple units in same lane have Conquer? (If replacing units)
- Does Conquer stack? (e.g., unit replaced had Conquer, new unit has Conquer = 2 bonus SP?) - Probably NO, since only winning unit matters

**Success Metric:** Playtesters prioritize lanes with Conquer units, fight to protect/kill them

### 2.5 üìç Deploy Restrictions (Empty Slots Only)
**Why Important:** Makes decisions more impactful. Can't freely replace units.

- [ ] **Enforce Rule**
  - Units can ONLY deploy to empty slots
  - If slot occupied by your unit, can't deploy there (unless Consume)
  
- [ ] **Consume Keyword**
  - Exception to rule: "Consume"
  - Units with Consume CAN deploy on top of other units
  - Acts like M1 sacrifice mechanic
  - Example: "Consume: Deploy: Deal damage equal to consumed unit's power"
  
- [ ] **UI Feedback**
  - Occupied slots are un-targetable (grayed out)
  - Hover shows tooltip: "Slot occupied"
  - Consume units CAN target occupied slots

**Cards to Implement:**
- A few Consume units (2-3 for testing)
- Test strategic impact: Does this create interesting decisions?

**Success Metric:** Playtesters think ahead about slot usage, occasionally "strangled" by full board

---

### 2.6 üîÑ Move Action
**Why Important:** Adds tactical repositioning. Counters bad matchups, enables positional synergies.

- [ ] **Move as an Action**
  - Moving is an ACTION that units can perform (not a trigger keyword)
  - Typically through activated abilities: "Activate: Move to a close empty slot"
  - Or granted by other cards: "Deploy: Move a close ally to an empty slot"
  
- [ ] **Movement Rules**
  - Can only move to adjacent empty slots (Close positioning)
  - Cannot move to occupied slots (respects deploy restrictions from 2.5)
  - Moving DOES trigger Deploy effects? (Test and decide)
    - If YES: Movement is powerful (re-triggers abilities)
    - If NO: Movement is pure repositioning
  
- [ ] **Visual System**
  - When activated, highlight valid move destinations (adjacent empty slots)
  - Movement animation (unit slides from slot to slot)
  - Clear indication that unit moved (brief glow/trail effect?)
  
- [ ] **Example Movement Abilities**
  - "Activate: Move this unit to a close empty slot"
  - "Deploy: Move a close ally to an empty slot" (enables ally to reposition)
  - "Activate: Swap this unit with a close ally" (if testing swapping)

**Cards to Implement:**
- 2-3 units with movement abilities for testing
- Test if movement is worth an activated ability slot
- Test different movement costs/restrictions

**Design Questions:**
- Does moving trigger Deploy effects? (Major balance implication)
- Should swapping be allowed, or only empty-slot moves?
- Is movement a quick action or main action?
- Do movement abilities cost charges, or are they unlimited per turn?

**Success Metric:** Playtesters use movement to reposition, dodge bad matchups, set up positional synergies (Close ally buffs)

**Success Metric:** Targeting flow is intuitive, no confusion about how to select targets
